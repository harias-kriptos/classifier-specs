🚀 Kriptos Base Genérica

Guía de Implementación para Cualquier Proyecto

Este documento explica cómo armar una base genérica reutilizable que funcione para cualquier proyecto en Kriptos, no solo el clasificador.

Objetivo: Una estructura de repo + workflow que hereden todos los proyectos nuevos, con skills, roles, configuración y MCPs listos para usar.

¿Por qué necesitamos una base genérica?

Actualmente el equipo tiene:

  • classifier-specs — POC que funciona para el clasificador
  • kriptos-engineering-standards — Template base de Cris
  • 5 Skills + Roles + Templates + Reglas dispersas

El problema: cada proyecto nuevo tiene que rearmar esto. La solución: un repo base que se copie y adapte.

Resultado esperado

Cuando un proyecto nuevo comience:

  1. DevOps clona el template
  2. Copia skills, roles, templates (reutilización directa)
  3. Adapta CLAUDE.md y estructura de carpetas
  4. Configura MCPs
  5. Equipo entra en Discovery (Fase 2a)
⏱️ Tiempo ahorrado: En lugar de 2-3 horas armando estructura, 30 minutos clonando y adaptando.

Visión General del Flujo

Las 4 Fases

# Fase Dónde Quién Output
1 Provisionar Monorepo GitHub DevOps Repo con .claude, skills, docs/context
2 Discovery + Contexto Claude Web Project PM, Arquitecto, Dev Decisiones, arquitectura, integraciones
3 Specs → TDD → Review Claude Web + Claude Code (híbrido) PM, Arch, Dev + Agente Features megeadas
4 Mantenimiento Contexto Claude Code (cada PR) Agente + Dev docs/context siempre alimentándose
⚠️ Importante: Las skills 01 (Brainstorm) y 02 (Spec + Threat Model) viven en Claude Web, no en Claude Code. Pero Claude Web tiene GitHub MCP, lo que le permite afectar al repo del producto (crear ramas, hacer commits con brainstorms y specs basados en templates).

Responsables: Dev asistido por agente
Dónde: Claude Code (Template Project)
Tiempo: 1-2 horas
Output: Markdown en docs/context/ + context/[proyecto]/

Proceso

3. Crea rama: git checkout -b docs/initial-context
4. Crea/edita:

7. Specs → TDD → Review (Fase 3)

Flujo híbrido: Claude Web (brainstorm + spec) → Claude Code (TDD + review)

Concepto clave: Las skills 01 y 02 (Brainstorm + Spec + Threat Model) ocurren en Claude Web de forma colaborativa (PM + Arquitecto + Dev). Claude Web tiene GitHub MCP y puede:
  • Leer templates del repo (templates/SPEC_TEMPLATE.md, etc)
  • Crear ramas (spec/KT-XXXXX-titulo)
  • Hacer commits con brainstorms y specs basados en templates
  • Abrir PRs para que Dev valide en Claude Code

🌐 Skills 1-2: En Claude Web (Colaborativo)

Skill 01: Brainstorm

Quién: PM + Arquitecto + Dev (Claude Web Project compartido)
Dónde: Claude Web con GitHub MCP + Jira MCP
Input: Ticket KT-XXXXX + docs/context/
Output: Commit en repo del producto: brainstorms/YYYY-MM-DD-KT-XXXXX-titulo.md 
# Flujo en Claude Web:

1. PM pega el ticket KT-12345 en Claude Web Project
2. Claude lee skills/01-brainstorm.md desde GitHub MCP
3. Claude lee templates/BRAINSTORM_TEMPLATE.md
4. Claude lee docs/context/ (servicios afectados)
5. Equipo conversa con Claude → Q&A iterativo:
   - ¿Qué problema resuelve?
   - ¿Qué supuestos hay?
   - ¿Qué riesgos identificamos?
   - ¿Qué preguntas quedan abiertas?
6. Claude crea brainstorm usando template
7. Claude (vía GitHub MCP):
   - Crea rama: brainstorm/KT-12345-export-reports
   - Hace commit: "docs(brainstorm): KT-12345 - export reports"
   - Abre PR para que Dev valide
8. Equipo revisa el PR → mergea a main

Skill 02: Spec + Threat Model

Quién: Arquitecto + Dev (con PM si hay dudas de producto)
Dónde: Misma conversación de Claude Web (continúa)
Input: Brainstorm anterior (ya en el repo)
Output: Commit en repo del producto: specs/YYYY-MM-DD-KT-XXXXX-titulo.md 
# Flujo en Claude Web (continuando):

1. Claude lee skills/02-spec-threat-model.md
2. Claude lee templates/SPEC_TEMPLATE.md (¡ya está definido!)
3. Claude lee templates/THREAT_MODEL_TEMPLATE.md
4. Claude lee el brainstorm aprobado
5. Arquitecto + Claude conversan:
   - Diseño técnico
   - APIs/contratos
   - Threat model (qué puede romperse, qué datos sensibles)
6. Claude crea spec usando template
7. Claude (vía GitHub MCP):
   - Crea rama: spec/KT-12345-export-reports
   - Hace commit: "docs(spec): KT-12345 - export reports + threat model"
   - Abre PR
8. Arquitecto + Tech Lead revisan PR → mergean
💡 Importante sobre Templates: Los archivos en templates/ son la estructura obligatoria. Claude Web los lee y genera el output respetando ese formato. Si quieres cambiar la estructura de una spec, cambias el template — no le dices a Claude que improvise.

Skill 05: Review + Evals

# Al abrir PR:

CI corre:
  ✓ pytest (tests pasan)
  ✓ ruff (lint)
  ✓ mypy (types)
  ✓ coverage >= 80%
  ✓ SonarCloud (cada dev con su token)
  ✓ Snyk (vulnerabilidades)

Claude Code/CI verifica:
  ✓ ¿Se cumple la spec?
  ✓ ¿Threat model resuelto?
  ✓ ¿Se actualizó docs/context/ si tocó arquitectura?
  ✓ ¿Commits siguen convención?

Si todo OK → ✓ Aprobado para merge
Si no → Feedback inline → vuelve a Skill 04

📋 Ejemplo de Flujo Completo (un ticket)

Día Skill Dónde Quién Output (en repo)
Lun 01 Brainstorm Claude Web PM + Arq + Dev brainstorms/2026-05-28-KT-1001-export.md
Mar 02 Spec + Threat Claude Web Arq + Dev specs/2026-05-29-KT-1001-export.md
Mar 03 Plan Claude Code Tech Lead + Dev plans/2026-05-29-KT-1001-export.md
Mié-Jue 04 TDD Claude Code Dev Branch + commits + PR
Vie 05 Review Claude Code + CI Reviewer + CI PR merged a main

Reglas para este flujo

  • Un ticket = una rama = un PR (por skill)
  • Nunca saltarse pasos (no sin brainstorm, no sin spec, etc)
  • Los 5 skills son lectura OBLIGATORIA antes de cada paso
  • Si el ticket toca servicios/arquitectura → actualiza docs/context/
  • PRs deben pasar CI (tests, linters, coverage 80%, SonarCloud, Snyk)
  • Formato de nombre: YYYY-MM-DD-KT-XXXXX-titulo.md
  • Templates son obligatorios — no improvisar estructura
✓ Fase 3 completada. Feature megeada. Código + brainstorm + spec + plan + docs en el repo.

8. Contexto Vivo (Fase 4)

Mantener docs/context/ siempre alimentado

Principio: Cada PR que toque arquitectura, servicios o integraciones DEBE actualizar docs/context/

Regla en CLAUDE.md

--- CONTEXT MAINTENANCE ---

Cada PR que toque:
  - Servicios (nueva integración, cambio de API)
  - Arquitectura (nuevo servicio AWS, cambio de flujo)
  - Dominios (nuevo concepto de negocio)

→ DEBE actualizar docs/context/

Ejemplo:
- PR agrega Lambda + SQS → Actualiza docs/context/architecture.md
- PR toca API externa → Actualiza docs/context/services/api-externa.md
- PR cambia flujo de pagos → Actualiza docs/context/ecosystem.md

Hook en .claude/settings.json

{
  "hooks": {
    "on-pr-created": {
      "trigger": "PR abierto",
      "message": "¿Este PR toca arquitectura/integraciones/servicios? Si sí, actualiza docs/context/"
    },
    "on-pre-merge": {
      "trigger": "Antes de merge",
      "check": "Si cambió stacks/ o context/, verifica que docs/context/ se actualizó"
    }
  }
}

Checklist en PR

Plantilla de PR (en repo):

## Cambios
- [ ] Feature nueva
- [ ] Bug fix
- [ ] Refactor
- [ ] Docs

## ¿Toca arquitectura/integraciones?
- [ ] Sí → Actualicé docs/context/
- [ ] No → No aplica

## Testing
- [ ] Tests nuevos o actualizados
- [ ] Coverage >= 80%
- [ ] Lint pasando (ruff)

## Checklist
- [ ] Commits siguen convención
- [ ] PR contra main
- [ ] Specs/Plan existen para este ticket

Ejemplo: Qué actualizar en docs/context/

Si el PR toca... Actualiza... Qué agregar
Nueva integración con AWS Kinesis docs/context/architecture.md Nuevo bloque: "Kinesis integration: evento X → Y"
Nuevo servicio externo (Stripe, SendGrid) docs/context/services/stripe.md API endpoints usados, formato de datos, errores
Cambio de flujo de pago docs/context/ecosystem.md Mapa actualizado con nuevo flujo
Nueva tabla en DB docs/context/architecture.md Schema, relaciones, índices importantes
Nueva variable de ambiente stacks/python-lambda/.env.example Y documentar en docs/context/

Frecuencia de revisión

  • Cada PR: El agente verifica que si cambió arquitectura, se actualizó docs/context/
  • Fin de sprint: Tech Lead revisa que docs/context/ esté actualizado
  • Mensual: PM + Arquitecto hacen audit de consistencia
✓ Fase 4 es continua. El contexto crece con el proyecto. Nunca queda stale.

9. MCPs (Integraciones Externas)

Cómo conectar Jira, Confluence, GitHub

MCP = Model Context Protocol: Protocolo para que Claude lea/escriba en sistemas externos
⚠️ Doble configuración: Los MCPs se configuran en DOS lugares:
  • Claude Web Project (para Skills 01-02): GitHub + Jira (+ Confluence opcional)
  • Claude Code (para Skills 03-05): mismos MCPs, configurados localmente en cada repo
Sin esto, Claude Web no podría hacer commits/crear ramas con brainstorms y specs.

MCPs Necesarios

MCP Qué hace Es obligatorio Quién lo configura
GitHub Leer repos, crear PRs, comentar issues ✓ Sí DevOps
Jira Leer tickets KT-XXXXX, actualizar status ✓ Sí DevOps
Confluence Leer/escribir docs de arquitectura ✗ No (opcional) Arquitecto si necesita

1. GitHub MCP

Configuración en .claude/settings.json

{
  "mcps": [
    {
      "name": "GitHub",
      "required": true,
      "auth": "oauth",
      "scope": [
        "repo",
        "read:user",
        "write:repo_hook"
      ],
      "settings": {
        "owner": "kriptos-io",
        "repo": "mi-proyecto-nuevo"
      }
    }
  ]
}

Qué puede hacer el agente

  • ✓ Leer archivos del repo
  • ✓ Ver commits, ramas, PRs
  • ✓ Crear PRs, comentar, sugerir cambios
  • ✓ Leer issues
  • ✗ Mergear PRs (debe hacerlo Dev)

2. Jira MCP

Configuración en .claude/settings.json

{
  "mcps": [
    {
      "name": "Jira",
      "required": true,
      "auth": "api_token",
      "env": {
        "JIRA_INSTANCE": "kriptos.atlassian.net",
        "JIRA_EMAIL": "dev@kriptos.io",
        "JIRA_API_TOKEN": "{{ cada-dev-genera-el-suyo }}"
      },
      "settings": {
        "project": "KT",
        "board": "KT Backlog"
      }
    }
  ]
}

Cómo generar token Jira (cada dev)

1. Ir a https://id.atlassian.com/manage-profile/security/api-tokens
2. Click: "Create API token"
3. Copiar token
4. En Claude Code: guardar en .env local
   JIRA_API_TOKEN=xxx
5. En settings.json: usar {{ JIRA_API_TOKEN }}

Qué puede hacer el agente

  • ✓ Leer tickets KT-XXXXX
  • ✓ Ver descripción, comentarios, attachments
  • ✓ Cambiar status (In Progress, Done, etc)
  • ✗ Crear tickets (Dev los crea, agente solo automatiza)

3. Confluence MCP (Opcional)

Configuración

{
  "mcps": [
    {
      "name": "Confluence",
      "required": false,
      "auth": "oauth",
      "settings": {
        "instance": "kriptos.atlassian.net",
        "space": "ENG"
      }
    }
  ]
}

Cuándo usarlo

  • Si Arquitectura vive en Confluence (ADRs, design docs)
  • Si tienes runbooks o troubleshooting guides ahí
  • Si colaboras con otros equipos que usan Confluence

Configuración Segura

⚠️ TOKENS:
- Nunca commitear tokens en el repo
- Usar .env.local o .env.example
- Agregar a .gitignore 
# .gitignore
.env.local
.env
credentials.json
                
# .env.example (sin valores reales)
JIRA_API_TOKEN=your-token-here
GITHUB_TOKEN=your-token-here
CONFLUENCE_TOKEN=your-token-here

10. Comandos Disponibles (.claude/commands/)

Los 5 comandos del pipeline

Qué es un comando: Un archivo markdown en .claude/commands/ que Claude Code ejecuta cuando Dev tipea /comando. Invoca automáticamente el skill correspondiente.
Importante: Skills 01 y 02 NO usan comandos locales. Se invocan directamente desde Claude Web cargando el archivo del skill. Los comandos en .claude/commands/ son para Skills 03-05 (locales en Claude Code). Aún así, los archivos de skills (skills/01-*.md y skills/02-*.md) viven en el repo para que Claude Web los lea vía GitHub MCP.

Los 5 Comandos

1️⃣ /brainstorm KT-XXXXX

Archivo: .claude/commands/brainstorm.md

---
name: brainstorm
aliases: [bs, storm]
description: Start Skill 01 (Brainstorm) for a ticket
usage: /brainstorm KT-12345
---

# /brainstorm

Invoca Skill 01 (Brainstorm) del pipeline.

**Qué hace:**
- Lee el ticket desde Jira
- Lee roles/product-manager.md
- Lee docs/context/
- Crea brainstorm en brainstorms/KT-XXXXX-titulo.md

**Cuándo usar:**
Cuando comienza un ticket nuevo. Primero del pipeline.

**Ejemplo:**
  /brainstorm KT-1001
  → Crea brainstorms/KT-1001-export-reports.md

2️⃣ /spec KT-XXXXX

Archivo: .claude/commands/spec.md

Invoca Skill 02 (Spec + Threat Model).

**Qué hace:**
- Lee el brainstorm anterior
- Lee roles/architect.md
- Lee stacks/[stack]/rules.md
- Crea spec en specs/KT-XXXXX-titulo.md
- Incluye threat model (seguridad, privacidad)

**Cuándo usar:**
Después del brainstorm. Segundo del pipeline.

**Ejemplo:**
  /spec KT-1001
  → Crea specs/KT-1001-export-reports.md

3️⃣ /plan KT-XXXXX

Archivo: .claude/commands/plan.md

Invoca Skill 03 (Plan + Atomic Tasks).

**Qué hace:**
- Lee la spec anterior
- Lee roles/tech-lead.md
- Crea plan con tasks atómicas
- Estima effort (story points)
- Crea plan en plans/KT-XXXXX-titulo.md

**Cuándo usar:**
Después del spec. Tercero del pipeline.

**Ejemplo:**
  /plan KT-1001
  → Crea plans/KT-1001-export-reports.md
  → Task 1: Setup database schema
  → Task 2: Create API endpoint
  → Task 3: Write tests

4️⃣ /implement KT-XXXXX

Archivo: .claude/commands/implement.md

Invoca Skill 04 (TDD Implementation).

**Qué hace:**
- Lee el plan anterior
- Lee roles/developer.md
- Dev implementa siguiendo TDD
- Escribe tests primero, luego código
- Abre PR contra main

**Cuándo usar:**
Después del plan. Cuarto del pipeline.

**Pasos:**
  1. /implement KT-1001
  2. Dev implementa localmente (TDD)
  3. Dev corre tests: pytest
  4. Dev hace commit y push
  5. PR abierto

5️⃣ /review KT-XXXXX

Archivo: .claude/commands/review.md

Invoca Skill 05 (Review + Evals).

**Qué hace:**
- Lee el código en el PR
- Verifica tests pasen
- Verifica coverage >= 80%
- Verifica linters (ruff)
- Verifica: ¿Se toca arquitectura? → ¿Se actualizó docs/context/?
- Si OK: aprueba. Si no: comentarios

**Cuándo usar:**
Cuando PR está abierto. Quinto del pipeline.

**Resultado:**
  - ✓ Si todo OK: Aprobado para merge
  - ⚠️ Si hay feedback: Dev arregla y vuelve a push

Anatomía de un Comando

---
name: comando-name
aliases: [alias1, alias2]
description: Una línea de descripción
usage: /comando ARG1 ARG2
requires: [JIRA, GitHub]
---

# /comando

Explicación de qué hace.

## Uso

```
/comando KT-12345
```

## Qué pasa

1. Paso 1...
2. Paso 2...
3. Paso 3...

## Resultado

Output esperado.

Cómo crear un comando nuevo

Si necesitas un comando personalizado (ej: /deploy, /migrate):

1. Crear archivo: .claude/commands/mi-comando.md
2. Definir frontmatter (name, aliases, description, etc)
3. Documentar qué hace
4. Dev invoca: /mi-comando
5. Claude Code lee el archivo y ejecuta

Ejemplo:
  /deploy [staging|prod]
  /migrate [up|down]
  /generate-report [html|pdf]
