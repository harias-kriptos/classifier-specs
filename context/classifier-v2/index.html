Orden recomendado para presentar

1. presentacion-clasificador — el pitch → 2. avance — dónde estamos → 3. architecture — el cómo técnico → 4. status — drill-down · DevOps: devops-reunion

Presentaciones

🎤

Pitch / método presentación

presentacion-clasificador.html

Deck narrativo: el método (spec-driven con agentes), el trabajo y el resultado. Sin números de ticket, pensado para contar la historia.

Audiencia: liderazgo / no técnicos  ·  Cuándo: abrir una reunión, explicar el porqué
Abrir
📈

Avance presentación

avance.html

Dónde estamos hoy: los 3 monorepos y qué incluye cada uno, estado global por épica, la fase en curso (Máquina de Estados), y specs/brainstorm/tickets por épica. Ésta es la principal para mostrar progreso.

Audiencia: liderazgo + equipo  ·  Cuándo: reportar avance
Abrir
🗺️

Arquitectura técnica

architecture.html

El diagrama end-to-end (Mermaid): Discovery → Máquina de Estados → GSE + Validación + LLM, con leyenda de estados y tarjetas de tickets por épica.

Audiencia: técnicos / DevOps  ·  Cuándo: explicar el flujo y las piezas
Abrir
📋

Status (drill-down) técnica

status.html

Estado de implementación granular: tablas de tickets por épica, modelo de infra, tabla JDC, gaps. Más detalle que avance.html.

Audiencia: equipo / PM  ·  Cuándo: revisar ticket por ticket
Abrir
⚙️

DevOps técnica

devops-reunion.html

Vista para DevOps: 1 monorepo por módulo, modelo de infra dentro del entregable, qué se provisiona en cada CloudFormation.

Audiencia: DevOps  ·  Cuándo: coordinar infra / repos
Abrir
📊

Dashboard técnica

dashboard.html

Tablero de estado de alto nivel (cards + flujo + próximos hitos). Se solapa con avance/status — ver nota abajo.

Audiencia: equipo  ·  Cuándo: vistazo rápido

Nota: solapamiento

avance.html, status.html y dashboard.html muestran "estado" desde ángulos parecidos. Recomendación: usar avance.html como la oficial de progreso (la más completa), status.html para el drill-down ticket a ticket, y archivar o borrar dashboard.html para evitar ruido. Decime y lo limpio.
