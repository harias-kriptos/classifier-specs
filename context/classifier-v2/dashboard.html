Classifier Backend — Tickets & Arquitectura

Vista consolidada del estado del backend del Classifier de Kriptos. Fase 1 unificada (scan + match + validación) + Fase 2 (GSE sample collection) con DDB compartida.
Última actualización: 2026-05-23 (consolidación post-refresh) · Owner backend: Haroldo Arias · Épica: KT-16369

📌 Resumen del flujo

Fase 1 cubre tres pasos secuenciales: scan de archivos (agente recorre el filesystem) → match de keywords (backend EMR Spark) → validación humana (cliente revisa candidatos en la Plataforma Web y aprueba/rechaza). El output de la validación dispara Fase 2.

Fase 2 es la recolección de samples (GSE): el agente extrae chunks de cada archivo validado, los sube a S3, el Anonymizer los procesa, y el LLM finalmente clasifica el cycle completo.

State machine único: una sola DDB classifier-cycles-state (KT-17009) mantiene el estado de enterprises + stations a lo largo de las dos fases.

Validation mode ∈ {enterprise, station}: el cliente confirma el set completo o station por station. KEM es la verdad absoluta para saber cuántas stations hay activas por enterprise.

✅ Fase 1 cerrada — 11 decisiones tomadas (10 técnicas + 5 externas resueltas). Estados del CYCLE renombrados (scanning → stations_complete → confirmed → phase2_collecting → complete). SNS topic phase1-ready-for-validation dropped — UI hace polling/subscription GraphQL. AppSync resolver confirmado como canal de validation. Ver DECISIONS-FASE-1.md + graphql-schema-appsync.md.

✅ Fase 2 cerrada — 9 decisiones técnicas + 4 externas con stubs. STATION row reusada entre F1↔F2 (UPDATE no PUT). Las 4 externas (canales Signal Handler/Anonymizer/LLM + WAF) no bloquean Skill 04 — funcionan con stubs.

🎯 Decisiones del refresh (consolidadas)

Fase 1 unificada
Scan + match + validación humana son una sola fase. No hay "Fase 1.5". La validación es el último paso de Fase 1 antes de pasar a Fase 2 (GSE).
DDB única classifier-cycles-state
Un solo state machine para todo el pipeline. PK = enterprise_id, SK con multi-prefix (CYCLE#, STATION#, REQUEST#). Stream NEW_AND_OLD_IMAGES alimenta a las 3 state lambdas.
Validation mode = enterprise OR station
Cliente decide si confirma el set completo del enterprise o station por station. Fase 2 corre en consecuencia (un manifest o N manifests).
KEM = verdad absoluta
Para stations_expected. Consultado en cycle creation (Fase 1, indexer). En Fase 2 cycle-init no se reconsulta — se mantiene el valor del CYCLE existente.
Mantener bucket crown_jewels
KT-16728 ya lo creó. Documentar semánticamente que contiene candidatos pre-validación. Sin rename in-place.
Validación por carpeta + keyword
Con override por archivo. Volumen alto (10k+ matches) hace inviable archivo-por-archivo. UX scaleable.
Multi-trigger gse-cycle-init
Desde el inicio. Distintos event sources → distinto process_type via env var. Reusable para classification y otros futuros.
Normalización compartida
normalize_category.py determinístico entre generador de keywords (IA) y matcher (joyas-priorizer). Versionar para evitar drift.
Aho-Corasick + pandas_udf
Para multi-token match de 2k-5k patrones × 10M+ archivos. Decisión final en Skill 02 de KT-16616.

🔄 Transiciones del CYCLE (state machine)

El CYCLE.status en la DDB compartida atraviesa estas transiciones secuenciales:
EstadoQuién lo seteaQuién consumeAcción
scanningcrown-candidates-indexer (al crear CYCLE)phase1-enterprise-barrierStations escanean en paralelo
stations_completephase1-enterprise-barrier (cuando stations_completed >= stations_expected)validation-mutation-handler (sigue permitiendo mutaciones), validation-confirm (habilita confirm)UI habilita botón "Confirmar todo". Cliente ya viene aprobando/rechazando desde antes.
confirmedvalidation-confirm (al confirmar global)gse-cycle-init (transiciona a phase2_collecting)Cliente confirmó. Sólo lectura.
phase2_collectinggse-cycle-init (al recibir manifest)gse-station-status, gse-enterprise-statusGSE recolectando samples per-station
phase2_skippedvalidation-confirm (si 0 approved)—Cycle termina sin disparar Fase 2
completegse-enterprise-status (cuando todas las stations cerraron)—Notify al LLM Process Queue

Cada state lambda **filtra** los stream records por el status correspondiente. Esto evita que `gse-station-status` procese stations de Fase 1, y que `phase1-enterprise-barrier` procese stations de Fase 2.
